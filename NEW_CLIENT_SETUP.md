# New Client Setup Guide

This project is a **reusable ZIMRA FDMS Bridge template**. The fiscalization engine, dashboard, schedulers, signatures and Supabase logging are all **done and intact**. For a new client you only need to:

1. **Register the device** with ZIMRA
2. **Match the invoice PDF template**

Everything else stays the same.

---

## What changes per client (only these)

| Item | Where | How |
|------|-------|-----|
| Device ID, activation key, serial | `.env` | from ZIMRA registration |
| TIN, VAT, company name | `.env` + `dashboard/src/config/company.js` | client details |
| Certificates | `certs/` | generated by `npm run setup` |
| Supabase project | `.env` (service key) + `dashboard/.env` (anon key) | new or shared project |
| Company branding on dashboard | `dashboard/src/config/company.js` | single file |
| Invoice PDF parsing | `src/utils/invoicePdfParser.js` | match client's invoice layout |
| Counter state | `C:\FDMS\state.json` | auto-created fresh |

**Everything in `src/` (fiscalization, receipts, signatures, schedulers), the API server, and the dashboard UI is reusable as-is.**

## Step-by-step

### 1. Clone / copy the project
```powershell
git clone https://github.com/Jkaseke1/RR-FDMS-.git client-name-fdms
cd client-name-fdms
npm install
cd dashboard ; npm install ; cd ..
```

### 2. Reset client-specific data
Clears the previous client's counters, logs, and certificates.
```powershell
npm run reset:client          # dry run - shows what will be removed
npm run reset:client:confirm  # actually clears it
```

### 3. Configure the backend
```powershell
copy .env.template .env
```
Edit `.env` and fill in:
- `FDMS_DEVICE_ID`, `FDMS_ACTIVATION_KEY`, `FDMS_DEVICE_SERIAL_NO`
- `CUSTOMER_TIN`, `CUSTOMER_VAT`, `CUSTOMER_NAME`
- `SUPABASE_URL`, `SUPABASE_SERVICE_KEY`
- Set `NODE_ENV` and `FDMS_BASE_URL` (test vs production)

### 4. Configure the dashboard
Either edit `dashboard/src/config/company.js` directly, **or** copy the env template:
```powershell
copy dashboard\.env.template dashboard\.env
```
Fill in `VITE_*` values (company name, device ID, TIN, VAT, Supabase URL + anon key, environment).

### 5. Register the device with ZIMRA
```powershell
npm run setup          # generates keys + certificate
npm run device:status  # confirm registration
```
This creates the certificate files in `certs/`.

### 6. Open a fiscal day
```powershell
npm run openday
npm run status
```

### 7. Match the invoice PDF template
Open `src/utils/invoicePdfParser.js` and adjust the parsing rules to the client's invoice layout:
- Currency detection labels
- Customer name location / skip list
- Line item columns
- Totals (Excl / Incl / Tax)

Test against a sample invoice:
```powershell
# drop a sample PDF into C:\FDMS\unsigned\
npm run pdf:process
```

### 8. Run the system
```powershell
npm run api          # backend API (port 3001)
npm run pdf:watch    # watches C:\FDMS\unsigned\ and fiscalizes
cd dashboard ; npm run dev   # dashboard (port 5173)
```

For production deployment as a Windows service, see `SERVER_SETUP_GUIDE.md`.

## Deploying the dashboard to GitHub Pages
1. Update `GITHUB_REPO` in `dashboard/vite.config.js` to the new repo name.
2. Push to GitHub.
3. Repo Settings → Pages → Source → **GitHub Actions**.
4. The workflow in `.github/workflows/deploy-dashboard.yml` auto-deploys on push.

## Quick reference - npm scripts
| Command | Purpose |
|---------|---------|
| `npm run setup` | Register device + generate certs |
| `npm run reset:client` | Dry-run clear of client data |
| `npm run reset:client:confirm` | Clear client data for real |
| `npm run openday` / `npm run closeday` | Open / close fiscal day |
| `npm run status` | Fiscal day status |
| `npm run pdf:watch` | Watch + auto-fiscalize PDFs |
| `npm run pdf:process` | Process pending PDFs once |
| `npm run api` | Start the dashboard API server |
| `npm run device:status` | Check ZIMRA device registration |

---

**That's it.** Register device + match the PDF template, and the client is live.
